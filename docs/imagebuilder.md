**Last revised:** 2026-04-27 (PR5 follow-up: Slurm-free base image policy)

## Slurm-free base image policy

**Slurm is installed at deploy time, never baked into the base image.**

When a node is deployed with a Slurm role (controller or worker), the clustr
deploy finalize step installs Slurm from the clustr-server's own bundled RPM
repository (`http://<clustr-server>:<port>/repo/el9-x86_64/`). This happens
inside the node's chroot before first boot. No Slurm packages are present in
the base image that is captured and staged on the clustr-server.

### Why

| Reason | Detail |
|---|---|
| **File conflicts** | If a base image contains Slurm RPMs (e.g. from OpenHPC 22.05.8) and the finalize step tries to install clustr Slurm 24.11.4, dnf fails with file conflicts on `/usr/sbin/slurmctld`, `/usr/sbin/slurmd`, etc. This is what PR5 lab validation hit. |
| **Version coupling** | Baking Slurm into the image ties the image to a specific Slurm version. Upgrading Slurm via a new clustr-server release would require rebuilding and re-staging every base image. |
| **Image size** | Slurm + MUNGE add ~80 MB to the squashfs. Not a constraint today, but worth avoiding. |
| **Simplicity** | The deploy-time install path is already exercised every time a node is deployed. Making the base image a dependency of that path adds complexity for no benefit. |

### What this means for image builds

The kickstart template generated by `internal/image/isoinstaller/kickstart.go`
enforces this policy at image build time. The `%post` block runs:

```bash
slurm_pkgs=$(rpm -qa | grep -iE '^slurm' || true)
if [ -n "$slurm_pkgs" ]; then
  echo "ERROR: base image contains Slurm packages ..." >&2
  exit 1
fi
```

If this check fails during a VM-based image build, the installer exits 1 and
the image is not captured. This prevents broken images from entering the
image store.

### Role package lists

The role package definitions in `internal/image/isoinstaller/roles.go` do not
include any Slurm packages. The relevant roles are:

| Role | Slurm packages excluded |
|---|---|
| `head-node` | `slurm`, `slurm-slurmctld`, `slurm-slurmdbd` |
| `compute` | `slurm`, `slurm-slurmd`, `slurm-pmi` |
| `gpu-compute` | `slurm`, `slurm-slurmd`, `slurm-pmi` |

Tests in `internal/image/isoinstaller/kickstart_test.go` assert that none of
these packages appear in the generated kickstart for any of these roles.

### Base image version / tag

After this policy change, any base image built from the updated kickstart will
be Slurm-free. Images built before this change (identified by tag or build
date) may contain OpenHPC Slurm 22.05.8 and should be rebuilt.

To identify whether an existing image is affected, inspect the image squashfs
on the clustr-server:

```bash
# Mount the squashfs (requires root, adjust path as needed):
mount -o loop /var/lib/clustr/images/<image-id>.squashfs /mnt/img

# Check for Slurm packages in the RPM database:
chroot /mnt/img rpm -qa | grep -i slurm

# Unmount:
umount /mnt/img
```

If any output appears, the image contains Slurm and should be rebuilt via
`POST /api/v1/images/build` with the current kickstart template.

## Defense-in-depth: finalize-side OpenHPC strip

Even if an old base image containing OpenHPC Slurm reaches a node deployment,
the finalize step in `internal/deploy/finalize.go::installSlurmInChroot` will
attempt to remove the conflicting packages before installing clustr's RPMs:

```
dnf -y --setopt=tsflags=noscripts remove slurm-ohpc slurm-ohpc-* ohpc-slurm-*
```

This is a best-effort safety net, not a substitute for rebuilding the image.
The strip uses `--setopt=tsflags=noscripts` to avoid running RPM `%preun`
scriptlets that may fail in the chroot environment.

If the strip succeeds and the subsequent clustr Slurm install also succeeds,
the node will boot with clustr Slurm 24.11.4. A post-install version check
logs an ERROR and records an audit event if the installed version is
22.05.8 (indicating the strip or install did not fully succeed).

## Chroot DNS preparation

During finalize, the deploy chroot has no DNS by default — the chroot
environment is isolated from the host's network stack. The finalize step
prepares DNS before invoking dnf:

1. Copies `/etc/resolv.conf` from the deploy environment (PXE initramfs)
   into the chroot, giving dnf the same DNS resolver that the initramfs uses
   (the clustr-server IP, advertised via DHCP option 6).
2. Ensures `/etc/hosts` contains at minimum:
   ```
   127.0.0.1 localhost localhost.localdomain
   ::1 localhost localhost.localdomain
   ```
   This prevents rpm scriptlet failures from missing localhost resolution.

Even though the primary Slurm repo URL is an IP address (clustr-server, no DNS
needed), the DNS prep is always applied because dnf may resolve EPEL or other
dependencies, and future code paths will rely on it.
